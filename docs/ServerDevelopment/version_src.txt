O dolaczaniu informacji o zrodlach do serwera
==============================================

SPIS TRESCI
-------
  1. Tlo; server_version()
  2. Wytyczne (dla paczerow i operatorow publicznych serwerow)
  3. Skad wziac version_src.h
  4. Format pliku version_src.h

TLO
----------
W dzialajacej instancji serwera wbudowana funkcja
server_version(), gdy poda sie jej argument w postaci lancucha znakow,
zwraca meta-informacje o kompilacji serwera.

Argument-lancuch moze wskazywac jedna z dostepnych wartosci
lub grup wartosci, mianowicie,

     "major","minor","release","ext"
		skladowe numeru wersji
     "string"	polaczony lancuch wersji
     "features" lista glownych funkcji
     "options"	ustawienia options.h
     "options/DEFAULT_FG_TICKS"
		konkretne ustawienie options.h
     "make"	nadpisania zmiennych linii polecen make
     "make/CPPFLAGS"
		konkretna zmienna make
     "source"	informacje o systemie kontroli wersji

Wiele z tych wartosci mozna nadpisac w version_src.h - pliku, ktory
NIE jest wlaczony do systemu kontroli wersji, lecz tworzony jest
recznie przez osobe przygotowujaca dystrybucje wydania lub przez
opiekuna pakietu (jesli zrodla pochodza np. z RPM-a), i/lub generowany
automatycznie w Twoim srodowisku budowania, w zaleznosci od sytuacji.

Dla wartosci w grupach "source" i "make", poniewaz z koniecznosci
pochodza *spoza* zrodel serwera, version_src.h jest JEDYNYM
sposobem na wprowadzenie tych wartosci do Twojej kompilacji serwera,
a dokladna metoda bedzie zalezec od Twojej sytuacji.

Mozesz tez zechciec kontrolowac poziom dostepnych szczegolow,
pozostawiajac niektore grupy czesciowo lub calkowicie bez informacji.

(N.B. zawartosc grupy "options" jest okreslana przez
version_options.h, ktory jest generowany przez version_opt_gen.pl)

WYTYCZNE
----------
Jesli prowadzisz publiczny serwer LUB jestes opiekunem pakietu dla
jakiejs dystrybucji systemu operacyjnego (np. Debian, Red Hat) i
tworzysz pakiet zrodlowy lub binarny serwera LambdaMOO przeznaczony
do publicznego udostepnienia, prosimy o przestrzeganie
nastepujacych wytycznych:

(1) W przypadkach, gdy zrodla serwera zostaly znaczaco
    zmodyfikowane wzgledem oficjalnie wydanej wersji, tzn. wykraczaja
    poza ustawienie jednego lub wiecej makr w options.h,
    lancuch "ext" powinien zawsze byc niepusty.

(2) Podawanie wartosci w grupie "source" nie jest wymagane,
    ale jesli sa podane, powinny byc prawdziwe. Obecnie
    rozpoznawane/zdefiniowane sa nastepujace wartosci:

       "vcs" = system kontroli wersji, jedno z ponizszych:

	   "release" -- kod *jest* archiwum wydania (tarball)
	   "git" -- kod jest zrzutem z repozytorium Git
	   "cvs" -- kod jest zrzutem z repozytorium CVS
	   "unknown" -- informacje o zrodlach sa niedostepne

       "vcs_version" = numer wersji instalacji Git lub CVS
       "commit" = identyfikator commita Git (40-znakowa wartosc SHA1)
       "url" = URL publicznego repozytorium

Zauwaz, ze jesli version_src.h jest pustym plikiem, te wytyczne
zostana spelnione automatycznie, patrz nizej.

SKAD WZIAC version_src.h
--------------------------

(*) Jesli zrodla Twojego serwera pochodza z dystrybucji wydania, powinienes
    juz miec odpowiedni plik version_src.h.

(*) Jesli zainstalowales pakiet zrodlowy dostarczony z Twoja dystrybucja
    systemu operacyjnego, opiekun pakietu powinien albo juz stworzyc
    version_src.h, albo dostarczyc sposob na jego wygenerowanie.

W obu przypadkach Twoj version_src.h powinien pozostac adekwatny, dopoki
Twoje zmiany ograniczaja sie do roznych dostosowan dozwolonych w
options.h. Jesli wykroczysz poza to, powinienes usunac lub zmodyfikowac
ten plik.

(*) Jesli zrodla Twojego serwera sa zrzutem z repozytorium Git,
    to repozytorium powinno zawierac instrukcje, jak wygenerowac
    version_src.h z logow repozytorium.

Najprawdopodobniej bedziesz musial dowiazac symbolicznie ./verion_hook
do dostarczonego skryptu powloki (lub perla), ktory nastepnie mozesz
zmodyfikowac wedlug wlasnych potrzeb.

./version_hook, jesli istnieje i jest wykonywalny, jest wywolywany przez
GNUmakefile przy kazdym uruchomieniu GNU make w Twoim katalogu zrodlowym.
Jesli uzywasz innej wersji make, bedziesz musial wywolac version_hook w
inny sposob, np. za pomoca checkout hooka Gita -- ktory wprawdzie nie
moze przechwycic nadpisan zmiennych make, ale moze przynajmniej ustawic
przydatne wartosci dla grupy "source".


FORMAT PLIKU version_src.h
-----------------------
Ten plik jest mozliwie pusta sekwencja dyrektyw #define.

    #define VERSION_MAJOR <number>
    #define VERSION_MINOR <number>
    #define VERSION_RELEASE <number>
    #define VERSION_EXT "<string>"

aby nadpisac rozne skladowe lancucha wersji ustawione w version.c

   #define VERSION_MAKEVARS(DEF) \
      DEF(<varname>,"<value>") \
      ...

   #define VERSION_SOURCE(DEF) \
      DEF(<varname>,"<value>") \
      ...

aby okreslic zmienne, ktore pojawiaja sie w grupach "make" i "source".
Zauwaz, ze <varname> nie jest w cudzyslowie, a <value> jest. Pamietaj
tez o koncowych ukosnikach; kazde z VERSION_SOURCE i VERSION_MAKEVARS
powinno parsowac sie jako pojedynczy #define. Wreszcie, nazwa parametru
"DEF" jest dowolna, o ile jest powtorzona doslownie w wartosci makra,
jak pokazano.

Jesli version_src.h jest pustym plikiem, bedzie to tak, jakby plik
zawieral

   #define VERSION_EXT "+?_ad_hoc_??"
   #define VERSION_SOURCE(DEF)  DEF(vcs,"unknown")
